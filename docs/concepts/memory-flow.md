# Memory Flow in Engram

This document explains what happens from a user turn to a generated response.

## Overview

```
User message
     │
     ▼
add_turn("user", ...)
     │
     ├─► Working memory (always)
     ├─► Neural coordinator feed (if enabled)
     └─► Ingestion decision
              │
              ├─► Episodic memory (if surprising/important)
              └─► Semantic memory (if preference/fact detected)

build_prompt(user_message, ...)
     │
     ▼
UnifiedRetriever ──► Working memory  (recent turns)
                ──► Episodic memory  (similar past episodes)
                ──► Semantic memory  (facts, preferences, typed relations)
                ──► Cold storage     (archived episodes, FTS5 fallback)
                ──► Neural hint      (familiarity/novelty signal, if enabled)
     │
     ▼
Token budget enforcement + pressure valve
     │
     ▼
Assembled prompt → LLM → reply

add_turn("assistant", reply)
     ├─► Working memory
     └─► Neural coordinator feed
```

## Step 1: Ingestion decision

When you call `add_turn("user", text)`, the ingestion layer runs a heuristic
pass over the text and decides:

- **Should an episode be stored?** High-information, non-ephemeral exchanges
  are routed to episodic memory. Greetings, filler, and very short turns are
  not.
- **Is there a preference or fact?** Text matching patterns like "I prefer X"
  or "I work at Y" triggers a write to semantic memory.
- **Is the turn too familiar to store?** If the `SurpriseFilter` is active
  (requires logprob-capable engine), low-perplexity turns are skipped.

The ingestion decision runs synchronously and takes < 1ms.

## Step 2: Retrieval

`build_prompt()` calls `UnifiedRetriever`, which pulls candidates from all
four explicit layers:

**Working memory** — the most recent N turns, always included in full.

**Episodic memory** — semantic search via ChromaDB embeddings. Returns the
top-K most similar past exchanges. Scored by cosine similarity, recency decay,
and importance.

**Semantic memory** — graph queries against Kuzu:
- Facts and preferences linked to the project's User node
- Typed relations (PREFERS, USES, KNOWS_ABOUT) connected to Entity nodes
- Co-occurrence graph context (sentences containing related entities)

**Cold storage** — FTS5 keyword search over archived episodes. Used as a
fallback when episodic memory doesn't return enough candidates, or when the
episodic layer is not installed.

Candidates from all layers are normalised to a common 0–1 score, deduplicated
by near-duplicate detection, and reranked before being fitted to the token budget.

## Step 3: Token budget enforcement

The assembled context is divided into sections (working, episodic, semantic,
cold) according to `TokenBudget`. If the total exceeds the budget:

1. Cold storage content is trimmed first (lowest priority)
2. Episodic content is trimmed next
3. Semantic content is trimmed if necessary
4. Working memory is never trimmed (recent turns are always preserved)

If the full prompt including the user message still exceeds the engine's
context window, the **pressure valve** fires: it compresses the memory block
via `engine.compress_prompt()`. The user message is never dropped.

## Step 4: Neural signal (optional)

If the neural layer is enabled and warmed up (≥ 20 turns), `NeuralCoordinator`
computes a surprise score for the query. This is used:

- As a hint in the assembled prompt (when surprise ratio > 1.5× average)
- To set episode importance dynamically when `store_episode()` is called
- As an input to the forgetting policy via `weight_surprise`

The neural layer does not affect retrieval ranking directly in the current
implementation — it influences *what gets stored* and *how important it is*,
not *what gets retrieved*.

## Step 5: Forgetting

Over time, the forgetting policy archives low-value episodes from episodic
memory to cold storage. This keeps the ChromaDB collection small and fast.

The policy scores each episode on recency, importance, access frequency, and
(if neural enabled) the surprise score at storage time. Episodes below the
retention threshold and older than `min_age_days` are archived.

The policy runs on a daemon thread after every `auto_trigger_interval` new
episodes (default: 50). It never blocks the chat path.

## What `respond()` does differently from the manual loop

`respond()` combines all of the above steps into a single call and also:

- Logs the run to `ExperimentMemory` (start time, model, metrics, outcome)
- Feeds both user and assistant turns through the neural coordinator in the
  correct order
- Applies the importance floor when storing any episode generated by the turn

The manual `add_turn()` / `build_prompt()` / `engine.generate()` loop gives
you more control over each step but requires wiring these pieces yourself.
